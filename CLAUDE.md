# CLAUDE.md

## 项目架构认知

本项目是 UI 自动化测试项目（Selenium + Pytest），采用分层架构：

```
base/base_page.py          # 公共底层工具 - 点击、输入、等待等基础操作
pages/login_page.py        # Page Object 层 - 页面定位 + 元素
pages/inventory_page.py
common/keywords.py         # 业务关键字层 - 封装常用业务场景
common/test_data.py        # 测试数据层 - URL、账号等静态数据
testcases/test_*.py        # 测试用例层 - pytest 用例
conftest.py                # 全局 fixture 和钩子
```

## Karpathy 四原则

### 1. 先思考再编码

**不要猜测。不要隐藏困惑。主动暴露取舍。**

实现之前：
- 明确说出你的假设。如果不确定，先问。
- 如果有多种解释，全部列出——不要默默选一个。
- 如果有更简单的方案，直接说。该反驳时就反驳。
- 如果某件事不清楚，停下来。说出困惑在哪里。提问。

### 2. 简洁优先

**写最少代码解决问题。不写试探性代码。**

- 不实现未要求的功能。
- 不为一次性代码抽象。
- 不添加未要求的"灵活性"或"可配置性"。
- 不处理不可能发生的异常场景。
- 如果写了 200 行但 50 行能搞定，重写。

问自己："资深工程师会觉得这个太复杂吗？"如果是，简化。

### 3. 精准修改

**只动必须动的代码。只清理自己造成的混乱。**

修改现有代码时：
- 不要"优化"周边的代码、注释或格式。
- 不要重构没坏的东西。
- 匹配现有风格，即使你会用不同方式写。
- 如果发现无关的死代码，提一下——不要删除它。

你的修改产生了孤儿代码时：
- 删除由**你的修改**导致的未使用的 import/变量/函数。
- 除非被要求，否则不要删除已有的死代码。

检验标准：每一行修改都应该能追溯到用户的原始需求。

### 4. 目标驱动执行

**定义成功标准。循环验证直到达标。**

把任务拆解成可验证的目标：
- "加验证" → "先写无效输入的测试，再让它们通过"
- "修 Bug" → "先写能复现 Bug 的测试，再让它通过"
- "重构 X" → "确保重构前后测试都通过"

多步骤任务时，写出简要计划：
```
1. [步骤] → 验证: [检查项]
2. [步骤] → 验证: [检查项]
3. [步骤] → 验证: [检查项]
```

## 项目专属规范

### 分层调用规则

- **BasePage**：只放 Selenium 公共操作（`click_element`, `input_text`, `get_text`, `wait_for_element_visible` 等）。不要写业务函数。
- **Page 层**：放页面元素定位符 + 该页面特有的操作。元素定位统一用元组格式 `(By.XXX, "value")`。
- **Keywords 层**：封装跨页面的业务场景（如登录→跳转）。不在测试用例中直接调 Page 对象。
- **测试用例**：只调 Keywords 或直接调 BasePage。不出现元素定位符，不出现 `By.ID` 之类的导入。

### 命名规范

- **类名**：大驼峰 `LoginPage`, `CommonKeywords`
- **方法/变量**：下划线小写 `open_login_page`, `username_input`
- **测试函数名**：`test_<场景>`
- **定位符变量**：语义化名词 `login_btn`, `error_text`

### 测试用例编写规范

- 一个测试函数只测一个场景
- fixture 选择：
  - `logged_driver` — 需要登录态的场景
  - `login_driver` — 测试登录页各场景
  - 不要直接用 `global_driver`
- 元素文本用参数化或常量，不要硬编码断言值
- 失败截图由 conftest 自动处理，用例里不写截图逻辑

### 工作习惯

- **精准测试**：改代码后只跑强相关的测试用例来验证，不要每次都跑全量。除非用户明确要求，否则用 `-k` 过滤执行。例如只改 `inventory_page.py` 就跑 `-k "test_inventory or test_add_to_cart or test_remove_from_cart"`，不改 login 相关就不跑。
- **单用例修复验证**：如果用户仅关心某一条用例的执行情况（如用户直接贴了 `pytest test_inventory.py::TestInventory::test_sort_items -v --tb=short`），则改完代码后只跑这一条用例来验证修复是否成功，不跑其他无关用例。
- **不要删除语法类注释**：注释中如果有“语法”两个字，不要删除这段注释。

### 技能沉淀

- **interview_qa.md**：位于 `skills/interview_qa.md`，记录项目中真实遇到的自动化问题及面试回答话术。内容包括：Safari `<select>` 下拉框挂死、session 隔离 + XPath `//` vs `.//`、Safari NoSuchFrameException、Safari 启动连接失败等。
  - **触发条件**：仅在用户明确提出"面向面试总结"或"写出面试回答"等要求时，才将问题的排查过程整理成面试回答话术追加到该文件。常规修复 Bug 时不自动执行此操作。

### Python 与 Pytest 风格

- 类型注解：公共函数建议加类型提示（如 `def input_text(self, locator, text: str)`）
- 多行字符串用 `"""`，单行提示用 `#`
- 全局 fixture 统一放在 `conftest.py`，测试文件仅包含测试函数
- 测试数据不分散写在用例里，统一放在 `common/test_data.py`
- 命名规范：常量大写蛇形，变量小写蛇形
